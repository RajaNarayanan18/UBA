01 — Agent Roster

6 AGENTS + 1 SIDECAR

The result of principled consolidation. Every agent boundary is justified by a distinct capability, a distinct layer, or a non-negotiable enterprise constraint.

AGENT 01
INTAKE &
TRIAGE
Deterministic

Fingerprints all 5,000 Jenkins jobs, builds the full dependency DAG, scores every job 0–100 via weighted formula, assigns wave placement, computes composite context hash, and stores snapshot_v1. The single serialised entry point — no other agent acts until this completes.

Consolidates: Intake · Dependency Graph · Risk Scoring · Context Snapshot (initial)
AGENT 02
KNOWLEDGE
REGISTRY
Hybrid

Manages both the plugin mapping registry and shared library AzDO template registry. LLM proposes mappings; humans ratify APPROXIMATE and NONE entries. PR-first design for library migrations. Libraries processed in descending order of dependent job count. One answer per plugin, globally consistent.

Consolidates: Plugin Registry · Shared Library Agent
AGENT 03
MIGRATION
Full LLM

Two-phase reasoning agent. Phase 1: resolves semantic conflicts between config.xml and Jenkinsfile AST JSON (never raw Groovy) to produce the Migration Brief. Phase 2: reads its own Brief, applies registry lookups, selects conversion tier, generates AzDO YAML + Conversion Manifest. 75–85% token reduction via deterministic pre-processing.

Consolidates: Analyzer · Converter
AGENT 04
VALIDATOR
Hybrid

Three-plane validation in strict order. Plane 1: syntactic schema check (deterministic). Plane 2: security scanning — secrets, RBAC scope, forbidden constructs (deterministic — security controls must be 100% reproducible). Plane 3: semantic alignment — does YAML implement exactly what the Manifest claims (LLM). Max 2 automated fix cycles before human escalation.

Unchanged — planes cannot be merged
AGENT 05
DEPLOYMENT
Deterministic

The only agent with AzDO write access. Four independent pre-conditions before any write: Validator PASS, signed JWT verified, quota available, no active circuit breaker. Archives Jenkins config immediately before writing. 50 prod deployments/hr globally enforced at MCP Gateway. Idempotency check before every create_pipeline call.

Unchanged — sole execution agent
AGENT 06
ORCHESTRATOR
Deterministic

Central state machine — 15 states per job. Sequences all 6 agents. Owns nightly drift detection batch (scheduled task, not a separate agent). Circuit breaker at 8% wave failure / 4hr window. Routes human gate requests. Durable — restarts replay from last committed state. Must be fully deterministic: audit trails require exact reproducibility.

Consolidates: Orchestrator · Context Snapshot Agent (nightly batch)
SIDECAR — NOT A PEER AGENT
OBSERVABILITY
SIDECAR
Read-Only · No MCP Tools · No State Store Writes

Consumes all Service Bus events as a passive subscriber. Writes structured telemetry to Azure Data Explorer. Fires SLA breach alerts before they breach. Has no decision authority, no write access, no MCP tools. Making it a full agent would add an event loop and state for zero architectural benefit. It is instrumentation, not architecture.

02 — Layer Separation

THE INVIOLABLE
SEPARATION

The architecture's central law: no agent containing LLM reasoning may have write access to any external system. This is enforced at the MCP Gateway — not by agent self-policing.

Architectural Law LLMs produce recommendations. Deterministic execution agents produce effects. These responsibilities must never coexist in the same process boundary. The 6-agent design maps cleanly to 3 layers with 2 agents per layer — every layer has a clear owner, a clear input, and a clear output.

3
MIGRATION — LLM REASONING
Migration Agent (1 of 50 parallel instances)
↕
Critical DesignThe LLM never receives raw Groovy or raw XML. Deterministic parsers in the MCP tool server pre-process both sources to structured JSON. Token reduction: 15,000–40,000 → 2,000–4,000 per job (75–85% saving). Parsers handle syntax; LLM handles semantics.
3.1
Deterministic pre-processing (inside MCP, not LLM): config.xml parsed → structured JSON (triggers, params, retention, agent labels, SCM). Jenkinsfile Groovy AST parsed → structured JSON (stages, steps, parallel, credentials, @Library, input steps, post conditions).
3.2
Phase 1 — Migration Brief: LLM resolves semantic conflicts between two sources. Detects implicit behaviours. Reads plugin + library mappings from Knowledge Registry (lookup, never independent reasoning). Produces: stages[], credential_refs[], plugin_refs[], complexity_tier, conflict_log[], manual_intervention_items[].
3.3
BLOCKING items? → CONVERSION_BLOCKED, job routed to human queue, flow stops. No blocking items → Phase 2 proceeds immediately.
3.4
Phase 2 — YAML + Manifest: selects tier (Rules Engine $0 / Mid-tier $0.03–0.08 / Frontier $0.15–0.55). Applies semantic mappings. Produces candidate AzDO YAML + Conversion Manifest (every mapping decision: what Jenkins construct → what AzDO construct, with what confidence, on what assumption).
3.5
Context-overflow path: if job too complex for single LLM context window → chunked analysis by stage → stage-level briefs assembled by Orchestrator before YAML synthesis. Prevents silent truncation of large scripted pipelines.
4
VALIDATION — 3 PLANES
Validator Agent (1 of 20 parallel instances)
↕
4.1
Plane 1 — Syntactic (Deterministic): mcp.azdo.validate_yaml_schema(). Pass/fail with exact errors. No LLM. Must be 100% reproducible — same input always same result.
4.2
Plane 2 — Security (Deterministic — non-negotiable): regex scans for hardcoded secrets (AWS_SECRET, API_KEY, password=). RBAC scope violations. Forbidden constructs. Required steps check (SAST scan, artifact signing). A security control that "misses on one run" is a security gap — must be deterministic.
4.3
Plane 3 — Semantic Alignment (LLM): does the YAML implement exactly what the Conversion Manifest claims? Reading comprehension + logical consistency across two structured documents. Without the Manifest, this is "does this YAML look right?" — with it, this is "does this YAML implement exactly what was promised?"
4.4
Fix loop: FAILED_FIXABLE → structured fix instructions injected into CONVERSION_REQUESTED event → Migration Agent treats as new conversion with constraints. Max 2 automated cycles. 3rd failure → human escalation. FAILED_BLOCKED (security) → immediate security team escalation, job locked.
5
DEPLOYMENT — PIPELINE CREATION
Deployment Agent
↕
The Critical Momentmcp.azdo.create_pipeline() is called here — and only here. This is the only place in the entire architecture where an AzDO pipeline is created. Four independent gates must clear before this call executes.
5.1
Four pre-conditions checked independently: (1) Validator PASS confirmed in State Store, (2) signed human approval JWT cryptographically verified by MCP Gateway, (3) mcp.quota.check_and_decrement() — global <50/hr, (4) no active circuit breaker on current wave.
5.2
Archive before write: Jenkins config archived to Config Snapshot Store (WORM) immediately before any AzDO write call. This is the rollback source — exact config at migration time, never a later drifted version.
5.3
Idempotency check: queries AzDO for existing pipeline with same content hash. Already exists with DEPLOYMENT_SUCCESS? → returns existing ID, no-op. This check executes inside the MCP tool server — survives agent restart mid-operation.
5.4
Shadow deploy first: mcp.azdo.create_pipeline() on non-prod branch → run → artifact hash comparison against Jenkins baseline. Shadow PASS → production promotion. Shadow FAIL → human review queue.
5.5
Production deploy: mcp.azdo.create_pipeline() on production → Jenkins set to READ_ONLY (not decommissioned). Publishes DEPLOYMENT_SUCCESS. Every action logged to Immutable Audit Log by MCP Gateway before execution.
04 — Observability & Distributed Tracing

TRACE EVERY
DECISION

Every event carries a trace_id from INTAKE_PENDING to COMPLETE. Without this, you see that a job failed — not where and why across the distributed trace. In a Fortune 100 incident postmortem, that is not acceptable.

Correlation ID DesignEvery event carries: trace_id (immutable for entire job journey) · span_id (unique per agent execution) · parent_span_id (links to triggering event). These three fields make the entire migration history for any job queryable as a single distributed trace in Azure Monitor.
Trace Event Schema — Every Event In The System
trace_id
Immutable UUID assigned at INTAKE_PENDING. Never changes. Links entire job journey across all agents, all MCP calls, all human gates.
span_id
Unique per agent execution. A single Migration Agent invocation = one span. A Validator re-run after fix cycle = new span.
parent_span_id
Links to the span that triggered this execution. Enables full causal chain reconstruction: why did this agent run? Which event triggered it?
agent_id
Which of the 50 parallel Migration instances, 20 Validator instances, etc. processed this job. Essential for correlating agent-level performance degradation.
duration_ms
Agent execution time. LLM calls broken into: pre-processing, LLM call, post-processing. Token counts in/out. Model tier used.
inputs_hash
SHA-256 of all inputs to this agent execution. With outputs_hash, creates a verifiable computation record: these inputs produced these outputs.
outputs_hash
SHA-256 of all outputs produced. Stored in Immutable Audit Log. Enables tamper detection — any modification to stored outputs is detectable.

06 — RAG Feedback Loop

COMPOUNDING
INTELLIGENCE

LLMs have no memory between calls by default. Three levels of memory make the Migration Agent progressively smarter across all 5,000 jobs — without retraining the underlying model.

Critical: Poisoning PreventionWithout quality gates on indexed documents, a bad Conversion Manifest that passes validation but fails in production gets indexed and injected into future jobs — systematically degrading automation quality over time. This is the most dangerous unaddressed failure mode without this design.
LEVEL 01
In-Context
Already present from Wave 1. Full Migration Brief + prior decisions in prompt context. Job 3,000 can receive Manifest from a similar completed job as reference example — no retraining required.
LEVEL 02
RAG Store
Azure AI Search with vector indexing (text-embedding-3-large). Top 3–5 nearest neighbours by cosine similarity > 0.82. Below threshold: no injection. Prevents irrelevant examples from degrading output.
LEVEL 03
Registry Updates
Fix Analyser (lightweight, inside Knowledge Registry) detects generalisable patterns from human corrections. Proposes registry updates → human ratification → all future jobs get correct mapping with no LLM involvement.
QUALITY GATE — WHAT GETS INDEXED
Only index Conversion Manifests that: 1. Passed all 3 Validator planes without a fix cycle 2. Passed shadow deployment artifact hash matched 3. Completed 30-day observation HEALTHY status throughout Poisoning prevention: Job enters ROLLBACK_COMPLETE → flag its Manifest in index Flagged Manifests → removed from index automatically Retrieval quality score → sent to Migration Agent alongside examples Low quality score → Migration Agent ignores retrieval, uses base model Human correction loop: Human fixes Converter output → correction delta stored Fix Analyser detects pattern → proposes registry update Human ratifies → enters production registry Future jobs with same plugin → deterministic lookup, no LLM needed Expected automation rate progression: Wave 1 baseline 68–75% Level-2 After Wave 2 RAG +5–8pts from retrieved examples Registry updates +3–5pts from ratified corrections Realistic ceiling 82–87% with all three investments

09 — Disaster Recovery & Failover

SURVIVE
ANYTHING

A 12–18 month migration at Fortune 100 scale will experience at least one regional outage, one LLM API degradation, and one storage consistency event. Recovery behaviour must be specified — not discovered under pressure.

MIGRATION STATE STORE

Consistency: Strong (for state transitions)
Primary region: Co-located with Jenkins cluster
Secondary region: Azure paired region
Failover: Automatic — RTO < 5 min, RPO = 0
Optimistic lock etag: Defined behaviour across regions — secondary never accepts a transition if primary etag differs. No split-brain state.
Retention: 3 years minimum
EVENT BUS
Tier: Azure Service Bus Premium

Geo-DR: Secondary namespace mirroring
