Layer Separation: Reasoning · Orchestration · Execution
The inviolable separation that makes the system auditable and governable
⚠
Architectural Law: No agent that contains LLM reasoning may have write access to any external system. LLMs produce recommendations. Deterministic execution agents produce effects. These responsibilities must never coexist in the same process boundary.
Layer Contents Inputs Outputs External Write Access LLM Calls
Reasoning Analyzer, Converter, Shared Library Agent, Plugin Registry Agent (partial), Validator (partial) Structured JSON from State Store only Structured recommendation documents (Migration Brief, candidate YAML, Conversion Manifest) None Yes
Orchestration Orchestrator Agent, Event Bus, State Machine, Human Gates Events from all agents, human approval tokens Authorized execution commands with signed tokens Event Bus only None
Execution Deployment Agent, Decommission Agent, Intake Agent, Monitoring Agent Authorized commands + signed human approval tokens Execution results (success/failure + structured diagnostics) Yes — via MCP only None
⚡
The MCP Gateway is the single enforcement boundary between all agents and all external systems. It enforces the tool-agent authorization matrix at the network level — not at the agent level. An agent cannot grant itself permissions it was not provisioned for. The kill switch at the MCP Gateway disables all write-capable tool calls instantly across all 14 agents with a single toggle.

01
Intake Agent
Job fingerprinting, inventory, deduplication — the only entry point
Deterministic Core
+
Why It Exists
Without a dedicated intake agent, multiple downstream agents would each attempt to discover and read Jenkins jobs independently, causing race conditions, duplicate State Store entries, and API saturation. The Intake Agent is the single serialization point — it creates a canonical record for every job before any other agent acts on it. Its SHA-256-based job ID makes the entire system idempotent: re-running intake never duplicates work.
Why No LLM
Fingerprinting is hashing. Deduplication is set membership. Inventory is pagination. These are algorithmic operations with exact correct answers. An LLM would add 8–15 seconds of latency, non-determinism in output structure, and ~$0.02/job in cost — for zero quality improvement. This is the clearest example of where LLM misassignment hurts a system.
Input → Output Contract
Input: BULK_INTAKE trigger + jenkins_namespace + business_unit

Output per job: job_id = sha256(config.xml + canonical_name) · config_hash · shared_library_refs[] · plugin_refs[] · status: INTAKE_COMPLETE

MCP Tools: mcp.jenkins.list_jobs() · mcp.jenkins.get_job_config(job_id) · mcp.storage.create_job_record()
Failure Handling
  • 429Delegates to Rate Limit Governor — never self-retries
  • 404Job listed but config unavailable → INTAKE_FAILED, manual queue
  • DUPSame job_id exists → no-op, no error emitted
  • XMLUnparseable config.xml → INTAKE_FAILED_UNPARSEABLE event

02
Dependency Graph Agent
Builds the DAG, determines migration order, assigns wave placement
Deterministic Core
+
Why It Exists
Migrating jobs without understanding their dependency graph is how you take down production. If Job A triggers Job B which deploys to prod, migrating Job B before Job A means the trigger never fires in AzDO and deployments stop silently. This agent computes the topological sort of all 5,000 jobs, ensuring that leaf nodes migrate first and jobs with many dependents migrate last — minimizing blast radius at every wave.
Technical Design
Backed by Cosmos DB Gremlin API (graph database). Nodes = jobs. Edges = dependency types: DOWNSTREAM_TRIGGER · ARTIFACT_CONSUMER · SHARED_LIBRARY. Wave assignment = topological sort output intersected with risk score tier. Circular dependencies are detected with DFS cycle detection — offending job pairs auto-assigned to WAVE_5 (manual).
Wave Assignment Logic
Wave 0: 10 pilot jobs — zero dependents, risk score 0–20, hand-selected
Wave 1: ~200 — leaf nodes, risk 0–20, no shared library dependencies pending
Wave 2: ~800 — low dependent count (≤5), risk 21–40
Wave 3: ~2,000 — moderate dependents, risk 41–65
Wave 4: ~900 — high dependent count (>20), risk 66–85, critical path jobs
Wave 5: ~90 — unmigrateable, EXTREME risk, circular deps → manual
Why No LLM
Graph traversal and topological sort are solved computer science problems. Kahn's algorithm or DFS produces the correct wave assignment deterministically in O(N+E) time. An LLM cannot improve on this and would introduce non-determinism into migration ordering — which is a safety property, not a quality property.
Wave Gate Enforcement
The Orchestrator calls mcp.graph.get_wave_readiness(wave_id) before starting any wave. This returns whether all dependencies for the wave's jobs are already in COMPLETE status. Wave N never starts until Wave N-1 passes its 2-week health check — enforced at the Orchestrator state machine level, not at the agent level.
03
Analyzer Agent ✦ LLM
Combines Jenkins UI + Jenkinsfile into the Migration Brief — the most complex reasoning task
LLM Core
+
Why LLM Is Required Here
The Analyzer must resolve semantic conflicts between two sources: the Jenkins UI config.xml (triggers, parameters, retention, agent assignment) and the GitHub Jenkinsfile (stages, steps, conditions, credentials). A deterministic parser cannot resolve "UI sets SCM poll every 5 min; Jenkinsfile defines weekly cron — which wins and why?" That requires judgment about intent. It must also detect implicit behaviors: a post { always { junit '**/*.xml' } } block implies test result publishing even if no explicit publish step exists.
Critical Architecture Decision
The LLM never receives raw Groovy or raw XML. Both sources are pre-processed by deterministic parsers in the MCP tool server before reaching the LLM. The LLM receives structured JSON (~2,000–4,000 tokens), not source code (~15,000–40,000 tokens). This is the single most important cost and quality decision in the platform — it reduces token usage by 75–85% while increasing reliability because parsers handle syntax; the LLM handles semantics.
What Each Source Contributes
Jenkins UI (config.xml): Build triggers · Parameters with defaults · Retention policy · Agent/label assignment · Pre/post build actions (UI-configured) · SCM configuration · Branch filters (Multibranch) · Workspace cleanup

GitHub Jenkinsfile: Stage structure · Step commands · Parallel blocks · Environment variables · withCredentials / credentials() calls · @Library references · Input approval steps · Post conditions · options{} block · timeout/retry settings
Migration Brief Output Structure
stages[] each with steps · conditions · confidence
credential_refs[] unified across both sources
shared_library_refs[] with registry status
plugin_refs[] with AzDO equivalents from registry
complexity_tier: SIMPLE / MODERATE / COMPLEX
manual_intervention_items[] with severity: BLOCKING / WARNING
conflict_log[] with resolution decisions and rationale
Failure Modes
  • DYNJenkinsfile is single line calling a library method → route to Shared Library Agent first, analysis blocked until library migrated
  • SCRScripted pipeline (arbitrary Groovy) → elevated scrutiny flag, confidence capped at 0.70
  • 404Jenkinsfile not found in GitHub → fall back to UI-only analysis, JENKINSFILE_NOT_FOUND flag
  • CTXContext overflow (very large job) → split analysis by stage, stage-level briefs assembled by Orchestrator

04
Converter Agent ✦ LLM
Migration Brief → AzDO YAML + Conversion Manifest. Never reads Jenkins or GitHub directly.
LLM Core
+
Why LLM Is Required Here
Semantic mapping between Jenkins and AzDO pipeline models requires judgment. Should a Jenkins parallel block become strategy.matrix or a dependsOn fan-out? The answer depends on what the parallel stages are doing and how they relate. A rule engine can handle the 40% of jobs that use exclusively direct-mapped constructs — the remaining 60% require LLM reasoning about intent.
Three-Tier Conversion Strategy
Tier 1 — Rules Engine (no LLM, ~30–40% of jobs): 100% of constructs have DIRECT registry mappings. Template substitution only. Cost: $0.

Tier 2 — Mid-tier LLM (~40–50% of jobs): MODERATE complexity, all plugins mapped, some APPROXIMATE mappings. ~6,000 tokens. Cost: ~$0.03–0.08/job.

Tier 3 — Frontier LLM (~15–20% of jobs): COMPLEX/CRITICAL risk. Unmapped plugins, custom DSL, scripted pipelines, parallel interdependencies. ~10,000 tokens. Cost: ~$0.15–0.40/job.
Key Semantic Mappings
withCredentials([usernamePassword()]) → env vars from variable group reference (credential value never in YAML)
parallel { stage('A'){...} stage('B'){...} } → jobs with dependsOn or strategy.matrix (based on relationship analysis)
stash/unstash → PublishBuildArtifacts@1 + DownloadBuildArtifacts@0
when { branch 'main' } → condition: and: [branch: 'main']
input step → ManualValidation@0 task in deployment job
post { always { junit } } → PublishTestResults@1 in a always-running step
Conversion Manifest — The Audit Trail
Every Converter call produces a Conversion Manifest alongside the YAML. This is a machine-readable document that records every mapping decision: what Jenkins construct mapped to what AzDO construct, with what confidence, and on what assumption. The Validator uses this manifest as the ground truth for semantic validation — not just the YAML in isolation. Without the Manifest, validation is "does this YAML look correct?" With it, validation is "does this YAML implement exactly what the Manifest claims?"
Failure Handling
  • BLKBLOCKING manual intervention items → CONVERSION_BLOCKED, no YAML produced, human queue
  • LOWLow confidence items → YAML produced with # REVIEW_REQUIRED annotations, Validator gets elevated scrutiny flag
  • SYNLLM produces syntactically invalid YAML → retry ×2 with explicit error feedback in prompt
  • LIB@Library reference not yet MIGRATED in registry → conversion blocked, AWAITING_LIBRARY_MIGRATION state
05
Validator Agent — Hybrid
Three-plane validation: syntactic (det), security (det), semantic alignment (LLM)
Hybrid Core
+
Three Validation Planes
Plane 1 — Syntactic (deterministic): AzDO YAML schema validation via mcp.azdo.validate_yaml_schema(). Pass/fail, exact errors. No LLM.

Plane 2 — Security (deterministic): Pattern scanning for hardcoded secrets, RBAC scope violations, forbidden YAML constructs per Governance Policy Pack. Regex + rule engine. No LLM.

Plane 3 — Semantic Alignment (LLM): Does the generated YAML actually implement what the Conversion Manifest claims? This is reading comprehension + logical consistency checking across two structured documents. LLM required because the relationship between intent (Manifest) and implementation (YAML) is not checkable by rules.
Security Validation Detail
secret_pattern_hits: regex scans for AWS_SECRET, API_KEY, password= patterns in YAML values
rbac_scope_violations: any pipeline granting above policy-defined permission ceiling → FAILED_BLOCKED
forbidden_constructs: continueOnError: true on security steps, allowScripts: true without approval
required_steps_missing: SAST scan, artifact signing — if absent → FAILED_FIXABLE
Result Taxonomy
  • PASSAll planes pass → Orchestrator routes to Human Gate 2 (or shadow deploy if LOW risk)
  • FIXFAILED_FIXABLE → structured fix report sent to Converter Agent. Max 2 automated fix cycles.
  • HUMFAILED_HUMAN_REQUIRED → third fix cycle failure or ambiguous semantic issue → human queue
  • BLKFAILED_BLOCKED (security violation) → immediate escalation to security team, job locked
Why Hybrid (Not Fully LLM)
Schema validation and security scanning must be 100% deterministic and reproducible. A RBAC violation that an LLM "doesn't notice" on one run but flags on another is a security control gap. These checks are rules — they must execute identically every time. Only the semantic alignment check has genuine ambiguity requiring LLM judgment.
06
Deployment Agent
Only agent with AzDO write access. Maximum governance constraints. Intentionally throttled.
Deterministic Core
+
Why It Must Be Separate
The Deployment Agent is architecturally separated from the Validator for one reason: execution must never be conflated with reasoning. The Validator recommends. The Deployment Agent executes — but only after: (1) Validator PASS, (2) signed human approval JWT verified cryptographically, (3) deployment quota available, (4) no active circuit breaker for the wave. Four independent gates before a single AzDO write call.
Blast Radius Controls
50 production deployments/hr globally — enforced at MCP Gateway, not agent
1 production deployment per BU per active window — distributed lock in State Store
Shadow mode first — all jobs run shadow before production promotion
Jenkins stays READ_ONLY — never decommissioned until 30-day clean window
Idempotency Design
Before calling create_azdo_pipeline, the agent checks AzDO for an existing pipeline with identical content hash. If found with status DEPLOYMENT_SUCCESS → no-op, returns existing pipeline ID. This prevents duplicate pipeline creation on retry or re-processing events. The check is performed inside the MCP tool server — not by the agent — ensuring it happens even if the agent is replaced or restarted mid-operation.
Rollback Prerequisite
The Deployment Agent archives the Jenkins job's current config in the Snapshot Store immediately before making any AzDO write call. This ensures that if a production rollback is needed at any point in the 30-day window, the exact Jenkins configuration that was in place at migration time is available for restoration — not a later version that may have drifted.
07
Monitoring Agent
30-day post-deployment health tracking. Webhook-driven. Owns decommission eligibility.
Deterministic Core
+
Why Webhook-Driven, Not Polling
Polling 5,000 AzDO pipelines on a schedule means either (a) infrequent polls miss regressions for hours, or (b) frequent polls saturate the AzDO API. With AzDO webhooks, each pipeline completion pushes an event to the Monitoring Agent's Service Bus subscription. The agent processes it in under 60 seconds. API calls are made only when there is data to process — O(run events), not O(pipelines × poll_interval).
Regression Thresholds
build_duration_delta: ±20% of Jenkins baseline average
test_pass_rate_delta: ±2 percentage points
artifact_hash_match: required for deterministic build jobs
consecutive_failures: ≥2 triggers REGRESSION_DETECTED
All thresholds are configurable per job risk tier via governance policy pack
Decommission Eligibility Criteria
  • 130+ consecutive days in HEALTHY status
  • 2≥5 production pipeline runs completed
  • 3Human decommission approval token present (Gate 3 — separate approver)
  • 4No open ITSM incidents linked to this job
  • 5All downstream dependent jobs already in COMPLETE state
Baseline Handling
Jenkins historical metrics are read from mcp.jenkins.get_baseline_metrics(job_id) at deployment time and stored in the Migration State Store. If Jenkins is already decommissioned before the 30-day window (rare, out-of-order scenario), the Monitoring Agent uses the first 3 AzDO runs as its own baseline, flags as INCONCLUSIVE, and continues observation with relaxed thresholds.
08
Orchestrator Agent
Central nervous system. Sequences all 14 agents. Never converts or deploys anything.
Deterministic Core
+
Why It Must Be Deterministic
The Orchestrator makes decisions that affect all 5,000 jobs — wave starts, circuit breaker triggers, rollback sequences, human gate routing. These decisions must be exactly reproducible and auditable. "The LLM decided to start Wave 3" is not acceptable in a regulated organization's incident postmortem. A rule-based state machine produces the same transition for the same inputs every time, and every transition is logged with full context.
State Machine (per job — 15 states)
INTAKE_PENDING → INTAKE_COMPLETE → RISK_SCORED → WAVE_ASSIGNED
→ AWAITING_LIBRARY_READINESS → ANALYSIS_QUEUED → ANALYSIS_COMPLETE
→ [HUMAN_GATE_1] → CONVERSION_QUEUED → CONVERSION_COMPLETE
→ VALIDATION_QUEUED → VALIDATION_COMPLETE → [HUMAN_GATE_2]
→ SHADOW_DEPLOYMENT → SHADOW_VALIDATED → PRODUCTION_DEPLOYMENT
→ OBSERVATION_WINDOW → [DECOMMISSION_QUEUED | ROLLBACK] → COMPLETE
Circuit Breaker Logic
If >8% of jobs in an active wave fail (any failure type) within a 4-hour rolling window: wave is paused instantly, deployment quota set to 0, ops team paged via PagerDuty. Wave cannot resume without a human explicitly re-opening it via the governance portal with a signed investigation-complete token. This is enforced at the Orchestrator state machine — no agent can work around it.
Durability Design
The Orchestrator is a stateless process backed by a durable state store (Cosmos DB). If the Orchestrator process crashes, it restarts and replays from the last committed state. State transitions use optimistic locking: a transition from state A to state B is rejected if the job's current state is not A. This makes all event processing idempotent — duplicate events trigger rejected transitions, not duplicate work.

Shared Library Agent ✦ LLM
Migrates Groovy shared libraries to AzDO templates. One of the highest-leverage agents in the system.
LLM Scale
+
Why This Agent Changes Everything
In a 5,000-job enterprise, @Library('platform-utils') may appear in 400 jobs. Without this agent, all 400 Converter instances independently try to understand what the library does and produce 400 potentially different translations. You cannot detect the inconsistencies. A platform failure in any one of them cascades to all 400 consumer jobs. With this agent, the library is migrated exactly once, human-reviewed, registered, and all 400 Converter instances reference the approved template by ID — a single source of truth.
Why LLM Is Required Here
Groovy shared libraries define custom DSLs. myPipeline.deploy(env: 'prod', strategy: 'blue-green') is not a standard Jenkins construct — it's a method defined in a library that internally calls dozens of Jenkins steps. The LLM must understand the library's functional intent from its Groovy implementation and produce an AzDO YAML template that achieves the same outcome. No parser can do this. No rules engine can handle arbitrary DSL semantics.
PR-First Design
Shared library migrations are never auto-merged. The agent creates a PR in the AzDO template repository. A platform engineer reviews and merges it. Only after merge does the registry entry update to MIGRATED. This is non-negotiable — a bad library migration affects hundreds of downstream jobs simultaneously. The PR review is the human gate before the highest-blast-radius action in the system.
Processing Order
Libraries are processed in descending order of dependent job count. The library used by 400 jobs migrates first. The library used by 2 jobs migrates last. This minimizes the Orchestrator's AWAITING_LIBRARY_MIGRATION queue depth and maximizes the number of jobs that can proceed to conversion at any given time.

Plugin Registry Agent — Hybrid
Maps Jenkins plugins to AzDO equivalents once. Every Converter and Analyzer reads from this registry.
Hybrid Scale
+
The Registry Principle
Without a central registry, you get N×M inconsistency: N Analyzer instances and M Converter instances each independently reason about what cobertura-plugin v1.16 maps to in AzDO. You get N+M different answers. The registry ensures one answer per plugin, ratified by a human, available to all agents instantly. A Converter Agent that reads from the registry never reasons about plugin mappings — it reads a lookup table.
Human Ratification Requirement
The LLM component of this agent discovers and proposes mappings. It does not ratify them. Every APPROXIMATE and NO_EQUIVALENT mapping must be reviewed by a platform engineer before entering the production registry. DIRECT mappings (where the AzDO task name and behavior exactly match) are auto-ratified by a deterministic lookup against AzDO's published task catalog.
Registry Schema Per Plugin
plugin_id · version_range
mapping_type: DIRECT / APPROXIMATE / NONE
azdo_task_ref: e.g. SonarQubePrepare@5
confidence: 0.0–1.0
migration_notes: what changes in behavior
limitations: what the AzDO equivalent cannot do
ratified_by: engineer identity + timestamp
approved_workaround: for NONE mappings — policy decision
Impact on Automation Rate
A fully populated and ratified Plugin Registry increases Level-2 automation success by 3–5 percentage points across the 5,000 jobs. Every plugin with a confirmed DIRECT mapping in the registry is a Converter decision eliminated — replaced with a deterministic lookup. Pre-populating the registry for the top 50 plugins (by frequency across all jobs) before Wave 1 pays dividends across 80–90% of all jobs.

Context Snapshot Agent
Detects when Jenkins jobs change mid-migration. Prevents migrating a job that no longer exists as analyzed.
Deterministic Scale
+
The Drift Problem at Scale
A 5,000-job migration runs for 12–18 months in a live enterprise. During that window, engineering teams continue modifying Jenkins jobs. Without drift detection, you analyze Job X in Month 1 and deploy its AzDO equivalent in Month 4 — but Job X has been modified 12 times in the interim. You deploy Month 1 behavior to production. The Monitoring Agent detects a regression. The root cause is invisible because the Migration Brief no longer matches the live job.
Composite Context Hash
Drift is detected via a composite hash of all semantic inputs:

sha256(config.xml content)
sha256(Jenkinsfile content)
sha256(sorted shared_library versions)
sha256(sorted plugin_id:version list)

All four combined → context_hash. A library version bump without any config change still triggers drift detection — critical, because @Library('platform-utils@main') pointing to a new commit is functional drift even if the Jenkinsfile is unchanged.
Drift Response by Migration State
  • ≤ANAANALYSIS_COMPLETE or earlier: invalidate brief, re-queue for analysis from new snapshot
  • CNVCONVERSION_COMPLETE: invalidate YAML, rollback to analysis queue
  • SDWSHADOW_DEPLOYMENT: pause shadow run, re-analyze, re-convert, re-validate before resuming
  • PRDPRODUCTION_DEPLOYMENT (already live): POST_DEPLOY_DRIFT_ALERT — human decision required
Drift Rate Monitoring
If >10% of in-flight jobs experience drift within a week, the Orchestrator triggers a migration pause and formally requests a "configuration freeze window" from business units. You cannot reliably migrate a portfolio where more than 1 in 10 active jobs is being modified weekly. This is a program management signal, not just a technical one.

Rate Limit Governor Agent
Centralized API budget across all agents. Prevents retry storms. Single Redis-backed singleton.
Deterministic Scale
+
Why Centralized Rate Limiting Is Non-Negotiable
Without a centralized governor, each of your 50 Analyzer instances implements its own backoff strategy. When Jenkins returns HTTP 429, all 50 instances receive it simultaneously and each starts its own exponential backoff timer. The timers are not coordinated. They all retry simultaneously at their backoff interval, producing a retry storm that saturates Jenkins for hours. A centralized token bucket prevents this entirely: the first 429 drains the bucket, and all subsequent requests queue behind the Governor's single retry schedule.
API Budget Configuration
Jenkins API: 200 req/min (configurable per Jenkins cluster capacity)
GitHub API: 4,800 req/hr (authenticated PAT limit, shared across all agents)
AzDO API reads: 300 req/min per organization
AzDO API writes: 50 req/hr (governance constraint, not AzDO limit)
LLM API: configurable per model tier, per-minute token budget
Read / Write Budget Separation
Read-heavy operations (analysis, validation) and write operations (deployment) use separate API token budgets for the same API. A wave of 800 Analyzer instances reading from AzDO for validation purposes does not consume the write budget needed for the Deployment Agent. Without this separation, a busy validation wave can exhaust the AzDO API budget and block deployments for other business units — a non-obvious cross-wave blast radius.
Technical Implementation
Redis atomic DECR operations on per-API token buckets. Each bucket refills at the API's documented rate limit interval. A leaky bucket algorithm ensures no burst accumulation. The Governor exposes a synchronous request_token(api, endpoint) call — agents block on this call, never implement their own backoff. The Governor reads the Retry-After header from AzDO 429 responses and uses that exact value — not a guessed exponential backoff.

Risk Scoring Agent
Scores all 5,000 jobs pre-migration. Drives wave assignment, human gate thresholds, and deployment quotas.
Deterministic Scale
+
Why Deterministic Risk Scoring
Risk scoring must be consistent, auditable, and challengeable. If a business unit lead asks "why is my job in Wave 4?", the answer must be a precise formula: "your job scored 72/100 because it has 28 downstream dependents (25 pts), deploys to 3 production environments (20 pts), and has 2 unmapped plugins (10 pts), plus 1 prior incident (10 pts), plus 7 credentials (7 pts)." An LLM risk score cannot be explained this way. A weighted formula can.
Scoring Formula
downstream_dependent_count × 25% weight (max 25pts)
production_scope (# prod environments) × 20% (max 20pts)
unmapped_plugin_count × 15% (max 15pts)
credential_binding_count × 15% (max 15pts)
shared_library_dependency_count × 10% (max 10pts)
historical_incident_rate × 10% (max 10pts)
stage_step_complexity × 5% (max 5pts)
Score → Tier → Consequences
0–20: LOW → Wave 1 · No human gate · Auto-deploy allowed
21–40: MODERATE → Wave 2 · Gate 2 only · Auto-deploy allowed
41–65: HIGH → Wave 3 · Gate 1 + Gate 2 · Shadow only
66–85: CRITICAL → Wave 4 · All gates · Manual prod approval per run
86–100: EXTREME → Wave 5 · Full manual migration · No automation
Override Policy
A BU IT Lead can request a risk tier downgrade for a specific job. The request is reviewed by the Security Architect and Migration Program Director. Overrides are logged with rationale and audited quarterly. Overrides that correlate with subsequent rollbacks are flagged in the quarterly review — creating accountability for risk tier manipulation.

Decommission Agent
The slowest and most conservative agent by design. Archives Jenkins jobs, never deletes them.
Deterministic Scale
+
Why It Exists as a Separate Agent
Decommissioning 5,000 Jenkins jobs without dedicated ownership turns into a manual tracking nightmare. Without this agent, teams forget to decommission jobs, or decommission them too early, or decommission the wrong ones. The Decommission Agent makes this a tracked, governed process — every job in the decommission queue has a documented eligibility date, a human approval token, and a 5-criteria eligibility check run automatically.
Archive vs Delete Policy
The Decommission Agent calls mcp.jenkins.archive_job() — which sets the job READ_ONLY and removes it from the active build queue. It never calls delete. Archived Jenkins configs are retained for 1 year minimum. Physical deletion requires compliance sign-off and is out of scope for automated processes. This is a regulated organization requirement: you must be able to prove what your CI/CD pipelines did for audit purposes, even after migration.
Capacity Throttling
Maximum 20 decommissions per day globally. This is intentionally conservative. The cost of accidentally decommissioning a Jenkins job that a team still needs outweighs the cost of taking 10 extra days to retire a batch. The daily cap also prevents a large wave of decommissions from coinciding with the end of a business quarter — when finance and compliance teams are most sensitive to infrastructure changes.
Downstream Dependency Check
Before decommissioning Job A, the agent queries the Dependency Graph to verify that all downstream dependent jobs are in COMPLETE state. If Job B depends on Job A and Job B has not yet been migrated, decommissioning Job A removes its trigger — silently breaking Job B's pipeline. This check prevents cross-job blast radius at the decommission stage.

Storage Strategy
Seven distinct stores, each optimized for its access pattern. Large payloads in Blob; metadata in Cosmos DB.
Migration State Store
Azure Cosmos DB — Multi-Region · Partition: business_unit · Row: job_id
Per-job state machine state, all agent output references (versioned), human approval records, health summary. Large payloads (YAML, briefs) referenced by blob_ref — Cosmos documents stay <10KB. Optimistic locking via _etag on all writes. 3-year retention.
Config Snapshot Store
Azure Blob Storage — Versioned · Path: snapshots/{job_id}/v{N}.xml
Immutable per-version snapshots of Jenkins config.xml + Jenkinsfile. SHA-256 composite context_hash stored in metadata. Azure Blob versioning as secondary safeguard. Used by Context Snapshot Agent for nightly drift reconciliation. 1-year retention + compliance extension.
Dependency Graph Store
Azure Cosmos DB Gremlin API · Nodes = jobs · Edges = dependency types
5,000-node directed acyclic graph. Edge types: DOWNSTREAM_TRIGGER, ARTIFACT_CONSUMER, SHARED_LIBRARY. Supports topological sort, cycle detection, reachability queries. Wave assignment computed by graph traversal. Incrementally updated by Context Snapshot Agent on drift events.
Plugin / Library Registry
Azure Table Storage · Low write, very high read · Human-ratified entries
Plugin mappings (DIRECT/APPROXIMATE/NONE) and shared library AzDO template references. Versioned by plugin_id + version_range. All APPROXIMATE/NONE entries require human ratification before production use. Read by Analyzer and Converter on every job. 5-year retention.
Deployment Quota Store
Redis (Azure Cache) · Ephemeral · Atomic operations
Real-time API token buckets per (API × endpoint × time_window). Deployment quota counters per (business_unit × quota_type). Event deduplication cache (event_id → processed, 24hr TTL). Distributed locks for concurrent deployment prevention. Recreated from policy on restart.
Immutable Audit Log
Azure Monitor Log Analytics · Write-once · HMAC-chained
Every agent action, MCP tool call, state transition, and human approval. Fields: audit_id, timestamp, actor, job_id, action, inputs_hash, outputs_hash, approval_token_id, hmac_signature. Each record's HMAC includes previous record's HMAC — tamper detection. Credential fields auto-redacted. 3-year minimum retention.
Operational Dashboard Store
Azure Data Explorer · Write-heavy ingest · 90-day retention
Real-time metrics: jobs by state, wave progress, human queue depth, API utilization %, regression/rollback counts, DLQ depth. Not used for audit — the Immutable Audit Log is the audit record. This store powers the ops team's real-time visibility dashboard. KQL queries for alerting.
🔗
Versioned Lineage Tracking: Every agent output records which input version produced it. brief_v3 derived_from snapshot_v2 → yaml_v2 derived_from brief_v3 → validation_v2 validated yaml_v2 against policy_pack_v1.4.2. A downstream agent receiving yaml_v2 can determine whether it was produced from the most recent snapshot version with a single version comparison — no content re-hashing required. This is the mechanism that makes drift detection O(1) at the downstream agent layer.

Governance — Human Gates & Controls
Four mandatory human gates, circuit breakers, blast-radius limits, and audit controls
GATE 0 — Pre-Wave
Wave Authorization
Approver: Program Director + BU IT Lead. Required before every wave starts. Artifact: wave job list, risk distribution, dependency readiness report. SLA: 3 business days. Enforcement: Orchestrator blocks wave start until signed JWT received from ITSM. No automation override possible.
GATE 1 — Pre-Conversion
Migration Brief Review
Approver: Team Lead + Security Architect. Required for risk tier HIGH (41+) and CRITICAL/EXTREME. Bypassed automatically for LOW and MODERATE risk jobs. Artifact: Migration Brief with human-readable summary. SLA: 2 business days. Jobs wait in ANALYSIS_COMPLETE state until approved.
GATE 2 — Pre-Production-Deploy
YAML + Shadow Run Sign-Off
Approver: Team Lead + Change Manager (ITSM change record required). Required for all risk tiers MODERATE and above. Artifact: validated YAML diff, Conversion Manifest, shadow run comparison report. SLA: 1 day for MODERATE, 3 days for HIGH/CRITICAL. Shadow run must pass before gate opens.
GATE 3 — Decommission
Jenkins Archival (4-Eyes)
Approver: BU IT Lead — must be different person from migration approver (four-eyes control). Required for all jobs, no exceptions. Cannot be waived. Artifact: 30-day monitoring health report, incident history. SLA: 5 business days. Jenkins config archived for 1 year minimum before any deletion discussion.

LLM Token Budget & Cost Architecture
Structured pre-processing reduces token usage by 75–85%. Tiered model allocation maximizes quality per dollar.
💡
Core Principle: LLMs never receive raw Groovy or raw XML. Both sources are pre-processed by deterministic parsers in MCP tool servers. The LLM receives structured JSON. This reduces token consumption from 15,000–40,000 tokens per job (raw) to 2,000–6,000 tokens per job (structured) — a 75–85% reduction with a corresponding improvement in reliability because parsers handle syntax; LLMs handle semantics.
